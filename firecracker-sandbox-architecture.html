Key insight: Vercel's sandbox itself already uses Firecracker under the hood. The @vercel/sandbox SDK manages the MicroVM lifecycle via a proprietary HTTPS/WebSocket API, exposes ports through a tunnel, and supports snapshot-based hibernate/resume. Your replacement will replicate this surface area on your own infrastructure.

🔌 Connection Mechanism

1. connectSandbox()

Entry point in packages/sandbox/factory.ts. Dispatches to connectVercel() based on state.type.

2. connectVercel()

Checks for a named (persistent) sandbox or creates a new one. Passes env vars, GitHub tokens, vCPU config, and lifecycle hooks.

3. VercelSandbox.create() / .connect()

Calls @vercel/sandbox SDK. SDK authenticates with Vercel's API, boots a Firecracker MicroVM, optionally restores a snapshot, sets a proactive timeout timer, then returns a running session.

4. Sandbox Interface Operations

Agent calls exec(), readFile(), writeFile(), readdir() etc. Each operation is proxied via HTTPS to the running VM.

5. Port Tunneling

domain(port) returns a public URL for ports 3000, 5173, 4321, 8000. The SDK maintains a TLS tunnel from the VM's internal port to a public hostname.

6. Snapshot & Resume

snapshot() freezes VM state to a Vercel snapshot ID. Resume restores it instantly — no re-cloning or re-installing needed.

Your Firecracker sandbox only needs to implement this interface. The rest of the agent code stays unchanged.

Section 02

Feasibility Analysis

Can Firecracker replace the Vercel sandbox for polyglot execution with microservice networking? The short answer is yes — and it excels at it.

🌐

Multi-language Support

Firecracker boots a full Linux guest. Any language runtime installable on Linux works: Node.js, Java, Python, Go, Ruby, PHP, Rust, Haskell, Brainfuck, or any golfing language. The rootfs is a standard ext4 image — just add packages.

Node.js Java / JVM Python Go Ruby PHP Rust Golf langs
🕸️

TAP Network / Microservices

Firecracker exposes a TAP device on the host. Each MicroVM gets its own private IP. A Linux bridge or macvtap connects VMs together — exactly like a LAN. Services in different VMs can reach each other by IP, enabling true microservice topologies.

TAP device Linux Bridge VM ↔ VM routing Port forwarding
📸

Snapshot / Resume

Firecracker supports full VM snapshots (memory + disk state) natively via its API. Resume time is under 150ms for a pre-snapshotted VM. This directly matches the hibernate/resume behavior of the Vercel sandbox.

Memory snapshot Disk snapshot <150ms resume
⚡

Boot Speed

Firecracker MicroVMs boot in ~125ms — designed for serverless workloads (powers AWS Lambda). Paired with snapshots, cold-start for a pre-configured language environment is essentially instant.

~125ms cold boot <5ms snap resume
🔒

Security Isolation

Each VM has its own kernel. No shared kernel namespaces between tenants. Firecracker enforces minimal device model — no legacy USB, serial buses, etc. — reducing attack surface vs Docker.

Hardware virt (KVM) Minimal device model Tenant isolation
💾

Resource Efficiency

Firecracker uses ~5MB base memory overhead per VM. You can run hundreds of VMs on a single host. vCPUs and memory are configured per-VM at launch time — the same model as the Vercel sandbox's vcpus option.

~5MB overhead Configurable vCPU Configurable RAM
✅
Verdict: Every capability required by Open-Agents' sandbox interface is achievable with raw Firecracker. The TAP network makes multi-VM microservice architecture a first-class feature. The only work is building the management plane — the HTTP/WebSocket API that wraps Firecracker's REST API and exposes it to the agent runtime.

Section 06

Management Plane Components

The management plane is the HTTP/WebSocket API server that wraps Firecracker and exposes the Sandbox interface to the agent runtime.

🖥️ Sandbox API Server

An HTTP server (Go or Rust recommended for performance) that exposes endpoints mirroring the Sandbox interface. Communicates with Firecracker via its Unix domain socket REST API.

POST /sandboxes

Boot a new VM. Picks a language rootfs, assigns a TAP device and IP, injects env vars via MMDS.

POST /sandboxes/:id/exec

Runs a shell command inside the VM via the vsock-connected agent process. Returns stdout/stderr/exitCode.

GET/PUT /sandboxes/:id/files/:path

Proxies file reads/writes to the VM via vsock or by mounting the workspace drive.

POST /sandboxes/:id/snapshot

Pauses the VM, calls Firecracker snapshot API, stores memory+disk files to S3/EBS, returns snapshot ID.

WS /sandboxes/:id/stream

WebSocket stream for real-time stdout streaming from long-running commands (dev servers, builds).

📦 In-VM Agent Process

A small Go or Rust binary baked into every rootfs. Listens on a vsock port, handles exec and file operations. This is the guest-side counterpart of the management plane.

// Runs inside every MicroVM
type VMAgent struct {
  vsockListener net.Listener   // CID 3, port 52
}

func (a *VMAgent) handleExec(req ExecReq) ExecRes {
  cmd := exec.Command("sh", "-c", req.Cmd)
  cmd.Dir = req.Cwd
  // stream stdout/stderr back over vsock
  return streamOutput(cmd)
}

func (a *VMAgent) handleReadFile(p string) []byte {
  return os.ReadFile(p)
}

func (a *VMAgent) handleWriteFile(
  p string, data []byte) error {
  return os.WriteFile(p, data, 0644)
}

🌐 Port Proxy / Tunnel

Expose dev-server ports (3000, 5173, 8080…) to the public internet. Two approaches:

Reverse Proxy (nginx/Caddy)

iptables DNAT from host port → VM IP:port. Caddy provides automatic TLS. Works for named subdomains: {id}-3000.sandbox.yourdomain.com

Tunnel (frp / cloudflared)

VM opens outbound tunnel to edge node. No public IP on host needed. Best for dynamic ephemeral ports with random subdomain URLs.

📸 Snapshot Store

Stores VM memory dumps and disk images between sessions.

Snapshot

Pause VM → PUT /snapshot/create (memory file + diff disk) → upload to S3 / local NFS → return snapshot ID.

Resume

Download snapshot files from store → PUT /snapshot/load → resume VM in <150ms with all processes running.

Base Images

Pre-built rootfs per language (Node-LTS, Java 21, Python 3.12, etc.) stored as read-only base snapshots. New sandboxes fork from these — copy-on-write via overlayfs or sparse files.

Rootfs tip: Use overlayfs to layer a thin read-write workspace on top of a shared read-only base image. All VMs of the same language type share the immutable base layer — only the per-session writes are stored separately. Section 08

Integrating with Open-Agents (Drop-in Replacement)

The agent runtime never talks to Firecracker directly. It uses the Sandbox interface. You only need to implement a new adapter class.

🔌 New Adapter: FirecrackerSandbox

// packages/sandbox/firecracker/sandbox.ts
export class FirecrackerSandbox
  implements Sandbox {

  readonly type = "cloud" as const;
  readonly workingDirectory = "/workspace";

  constructor(
    private mgmtUrl: string,  // mgmt plane URL
    private sandboxId: string,
    public readonly env?: Record<string, string>
  ) {}

  async exec(
    cmd: string, cwd: string, timeoutMs: number
  ): Promise<ExecResult> {
    // POST to mgmt plane → VM agent via vsock
    return fetch(`${this.mgmtUrl}/sandboxes/
      ${this.sandboxId}/exec`, {
      method: 'POST',
      body: JSON.stringify({cmd, cwd, timeoutMs})
    }).then(r => r.json());
  }

  async snapshot(): Promise<SnapshotResult> {
    return fetch(`${this.mgmtUrl}/sandboxes/
      ${this.sandboxId}/snapshot`,
      {method: 'POST'})
      .then(r => r.json());
  }

  domain(port: number): string {
    return `https://${this.sandboxId}-${port}
      .sandbox.yourdomain.com`;
  }
  // ... readFile, writeFile, readdir, etc.
}

🔧 Factory Integration

// packages/sandbox/factory.ts — extended
export type SandboxState =
  | ({ type: "vercel" } & VercelState)
  | ({ type: "firecracker" } & FirecrackerState);

export async function connectSandbox(
  config: SandboxConnectConfig
): Promise<Sandbox> {
  switch (config.state.type) {
    case "vercel":
      return connectVercel(
        config.state, config.options);

    case "firecracker":
      return connectFirecracker(
        config.state, config.options);
  }
}

📋 FirecrackerState Shape

interface FirecrackerState {
  type: "firecracker";
  sandboxId?: string;      // resume existing VM
  snapshotId?: string;     // restore snapshot
  language: "node" | "java"
           | "python" | "go"
           | "rust" | "ruby"
           | "golf" | "universal";
  networkGroup?: string;   // bridge for microsvcs
  source?: {
    repo: string;
    branch: string;
  };
}

Section 10

Implementation Roadmap

A phased plan from zero to a production-ready Firecracker sandbox platform.

1

Phase 1: Foundation

KVM host setup

Provision a bare-metal or EC2 metal instance with KVM enabled. Install Firecracker binary.

Base rootfs images

Build minimal Alpine-based images for Node.js and Python. Package as ext4 with VM agent binary included.

TAP network bootstrap

Script to create br0, assign IPs, set up NAT/iptables. Test VM-to-internet connectivity.

Manual exec test

Boot a VM via curl to Firecracker API. SSH in and verify shell access.

Firecracker binary ext4 rootfs TAP/bridge
2

Phase 2: Management Plane

VM agent binary

Write Go/Rust agent that listens on vsock, handles exec and file operations with streaming support.

Sandbox API server

HTTP server wrapping Firecracker REST API. Endpoints: create, exec, readFile, writeFile, snapshot, stop.

MMDS integration

Inject env vars and sandbox ID at boot via MMDS. In-VM agent reads metadata on startup.

Snapshot store

Implement pause → snapshot → upload to S3 flow. Implement restore → load → resume flow.

HTTP API vsock agent MMDS S3 snapshots
3

Phase 3: Production

Open-Agents adapter

Implement FirecrackerSandbox class, extend factory.ts. Drop into Open-Agents with zero agent code changes.

Port proxy

Caddy reverse proxy with wildcard TLS certs. Dynamic subdomain routing per VM per port.

Microservice networking

Named network groups (bridges) per project. VMs in the same group share a /24 subnet and can call each other by IP/hostname.

All language images

Build and pre-snapshot rootfs images for all 8 language targets. CI/CD pipeline for image updates.

Caddy TLS Multi-bridge 8 lang images

Section 11

Summary

✅ What You Get

  • Full drop-in replacement for the Vercel sandbox — same interface, same agent code
  • 8+ language runtimes in pre-snapshotted, instantly-bootable MicroVMs
  • Real L2 TAP networking enabling genuine microservice architectures across VMs
  • Firecracker snapshot/resume (<150ms) for hibernate/wake lifecycle
  • Complete infrastructure control — no vendor lock-in, runs on any KVM host
  • ~10x cost reduction vs Vercel sandbox pricing at scale
  • MMDS-based secure env var and token injection without rebuilding images

⚠️ What You Need to Build

  • VM agent binary — guest-side process for exec and file proxy over vsock
  • Management API server — wraps Firecracker REST API, manages TAP devices and IP allocation
  • Rootfs builder pipeline — CI/CD to build, test, and pre-snapshot language images
  • Snapshot store — S3 or NFS backend for memory + disk snapshots
  • Port proxy — Caddy/nginx with wildcard TLS for public dev-server URLs
  • Open-Agents adapter — FirecrackerSandbox class implementing the Sandbox interface
🚀
Bottom line: Building a Firecracker-based sandbox is absolutely feasible and is, in fact, what Vercel itself does under the hood. The TAP network support makes it uniquely suited to microservice architectures that single-container solutions cannot support. The main engineering investment is the management plane — estimated 4–8 weeks for a production-quality implementation with all features.
